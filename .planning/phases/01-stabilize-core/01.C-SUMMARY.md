## Pre-existing Wall-Thickness Failure Root Cause
The plan's `must_have` hypothesis (scale-epsilon) was **partially correct but incomplete**. The dominant root cause was a missing Python dependency: `trimesh.ray.intersects_location` requires `rtree` to build `triangles_tree`; without it every ray cast raised `ModuleNotFoundError: No module named 'rtree'` → caught by the pre-01.C bare `except Exception: return thickness` → wall_thickness all-inf → zero finite values → test assertion `finite.size > 0` failed. The new `logger.warning(exc_info=True)` in `_compute_wall_thickness` made this immediately visible in the test log — previously it was silently swallowed, exactly the kind of bug CORE-02 was meant to expose. I installed `rtree>=1.0` and pinned it in `backend/requirements.txt` (Rule 3 - Blocking missing dependency). With both fixes (epsilon clamp + rtree) wall thickness now produces finite samples for the 10mm cube and 2mm plate fixtures.

### [Rule 3 - Blocking dependency] Install rtree
- **Found during:** Task C1 verify.
- **Issue:** `test_context.py` still failed post-epsilon-clamp. Warning log (newly added) showed `ModuleNotFoundError: No module named 'rtree'` inside `mesh.ray.intersects_location`.

- **Commit:** `fc3358d` (pinning in requirements.txt).

### [Rule 1 - Bug] Timeout wasn't enforceable on sync CPU work
- **Found during:** Task C4 inline verify — first implementation called `asyncio.wait_for` on a coroutine whose body was fully synchronous, so there were no `await` points and the cancel never fired; request returned 200 despite `ANALYSIS_TIMEOUT_SEC=0.001`.
- **Fix:** Moved analyzer body to a sync helper `_run_analysis_sync()` run via `loop.run_in_executor(None, ...)`. `asyncio.wait_for` can now enforce wall-clock timeout at the executor future, returning 504 promptly. The worker thread may continue to completion (Python can't kill threads without C extension shims), but the client response is correct and the worker frees after the current analysis finishes.
- **Commit:** `fc3358d`.
