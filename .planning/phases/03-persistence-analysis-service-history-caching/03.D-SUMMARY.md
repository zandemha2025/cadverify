# Phase 3 Plan D: Usage Dashboard UI Summary

**One-liner:** Dashboard page with cursor-paginated analysis history table, rate-limit quota display, and analysis detail view reusing AnalysisDashboard component.

## What Was Built

1. **API client functions** (`api.ts`): `fetchAnalyses` (cursor pagination + rate-limit header extraction), `fetchAnalysis` (detail fetch), plus `AnalysisSummary`, `AnalysisDetail`, `RateLimits` TypeScript interfaces.

2. **AnalysisHistoryTable** component: Paginated table with verdict badges (green/yellow/red), face counts, duration, relative timestamps. Verdict filter dropdown. 500-item client-side cap (T-03D-02 mitigation). Clickable rows navigate to detail view.

3. **QuotaDisplay** component: Progress bars reading X-RateLimit-Remaining/Limit headers. Green/yellow/red color coding at 50%/80% thresholds. Graceful "Quota data unavailable" fallback (T-03D-01 mitigation).

4. **Dashboard page** (`/dashboard`): Three sections -- quota consumption, recent analyses table. Wires rate limits from API responses to QuotaDisplay via state lifting.

5. **Analysis detail page** (`/dashboard/analyses/[id]`): Fetches analysis by ULID, renders metadata header + full result via existing AnalysisDashboard component. Back button, 404 handling.

## Known Stubs

- **API key activity section**: The plan specified an "API-key activity section shows list of keys with last-used timestamp and analysis count from usage_events". This requires a backend endpoint aggregating usage_events per key that does not yet exist. The dashboard page omits this section rather than rendering stubs. This is a known gap to be addressed when the usage_events aggregation endpoint is built.

## Threat Mitigations Applied

- **T-03D-01** (missing rate-limit headers): QuotaDisplay shows "Quota data unavailable" when headers absent.
- **T-03D-02** (infinite scroll memory): 500-item client-side cap with "Load more in new view" link.
