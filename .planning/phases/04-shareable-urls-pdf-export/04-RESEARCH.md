## 1. WeasyPrint Setup and Integration

### Installation and Dependencies

WeasyPrint renders HTML/CSS to PDF via Cairo + Pango. On a Debian/Ubuntu-based Docker image, system packages are required:

```bash
apt-get install -y libpango-1.0-0 libpangoft2-1.0-0 libcairo2 \
  fonts-dejavu fonts-liberation fonts-noto-core
```

Python package: `weasyprint>=62.0` (latest stable). WeasyPrint 62+ dropped Python 3.8 support and improved CSS Grid/Flexbox handling.

**Key constraints (Pitfall 12):**
- System fonts MUST be baked into the Docker image -- WeasyPrint crashes with `AttributeError: NoneType has no attribute 'family'` if no fontconfig fonts are found.
- Noto fonts cover Unicode engineering symbols (um, o, +/-) needed in DFM reports.
- Phase 6 Dockerfile must include the system package install -- Phase 4 only adds the Python dependency.

### API Usage Pattern

```python
from weasyprint import HTML
from jinja2 import Environment, FileSystemLoader

env = Environment(loader=FileSystemLoader("backend/src/templates/pdf/"))
template = env.get_template("analysis_report.html")
html_str = template.render(analysis=analysis_data)
pdf_bytes = HTML(string=html_str).write_pdf()
```

**Performance:** Typical text+tables PDF renders in 1-3 seconds. No external network calls needed (all CSS/fonts are local). Memory usage is moderate (~50-100 MB peak for a complex report).

### CSS for Print

WeasyPrint supports `@page` CSS rules for controlling page size, margins, headers/footers:

```css
@page {
  size: A4;
  margin: 2cm 1.5cm;
  @bottom-center { content: "Generated by CadVerify"; }
}
```

Color verdicts (green/amber/red) render in color but must remain legible in grayscale (use sufficient contrast, not just hue).

## 3. Base62 Short ID Generation

### Implementation

Base62 uses `[0-9A-Za-z]` (62 characters). For 12 characters:
- Entropy: 62^12 = 3.2 x 10^21 possible IDs
- Brute-force at 1M guesses/sec: ~101,000 years

```python
import secrets
import string

BASE62_CHARS = string.digits + string.ascii_letters  # 0-9, A-Z, a-z

def generate_short_id(length: int = 12) -> str:
    """Generate a cryptographically random base62 string."""
    raw = secrets.token_bytes(length)  # 96 bits of entropy
    result = []
    for byte in raw:
        result.append(BASE62_CHARS[byte % 62])
    return "".join(result)
```

**Note on bias:** Using `byte % 62` introduces slight bias (256 is not evenly divisible by 62). The maximum bias per character is `(256 % 62) / 256 = 0.07%` -- negligible for this use case. For zero bias, use rejection sampling, but the security margin (72+ bits of entropy) makes this unnecessary.

### Database Integration

The `analyses.share_short_id` column already exists (Phase 3, migration 0002) with:
- `Text, unique=False, nullable=True` in ORM
- `ix_analyses_share` partial unique index `WHERE share_short_id IS NOT NULL`

Share flow:
1. Generate short_id via `generate_short_id()`
2. `UPDATE analyses SET share_short_id = :id, is_public = true WHERE ulid = :ulid AND user_id = :user_id`
3. On collision (unique constraint violation), retry with new ID (probability: ~1 in 10^21)

Unshare flow:
1. `UPDATE analyses SET share_short_id = NULL, is_public = false WHERE ulid = :ulid AND user_id = :user_id`
2. Next `GET /s/{old_id}` returns 404 immediately (no cache to invalidate)

Next.js supports dynamic metadata via `generateMetadata()`:

```typescript
export async function generateMetadata({ params }: Props): Promise<Metadata> {
  const data = await fetchSharedAnalysis(params.shortId);
  return {
    title: `${data.filename} - DFM Analysis`,
    openGraph: {
      title: `${data.filename} - DFM Analysis`,
      description: `Verdict: ${data.verdict} | ...`,

  };
}
```

### Server-to-Server API Call

The server component fetches from the backend API at render time. This requires:
- `NEXT_PUBLIC_API_URL` or a separate `INTERNAL_API_URL` env var for server-side calls
- The backend `/s/{short_id}` endpoint returns JSON (not HTML) -- the Next.js server component renders the UI

## 5. Asyncio Semaphore for PDF Concurrency

### Pattern

```python
import asyncio

# Module-level semaphore -- per-process, adequate for single-instance beta
_pdf_semaphore = asyncio.Semaphore(2)  # Max 2 concurrent PDF renders per process

async def generate_pdf(analysis) -> bytes:
    async with _pdf_semaphore:
        # WeasyPrint is CPU-bound; run in executor to avoid blocking event loop
        loop = asyncio.get_event_loop()
        pdf_bytes = await loop.run_in_executor(None, _render_pdf_sync, analysis)
        return pdf_bytes
```

**Why run_in_executor:** WeasyPrint's `write_pdf()` is synchronous and CPU-bound (1-3 seconds). Running it directly in an async handler would block the entire event loop. `run_in_executor` offloads to a thread pool.

**Semaphore scope:** Per-process only. At multi-instance scale, promote to a Redis-based distributed semaphore (e.g., `aioredlock`). For single-instance beta, this is sufficient.

### Cache-First Pattern

```python
async def get_or_generate_pdf(analysis_ulid: str, session: AsyncSession) -> bytes:
    cache_path = Path(PDF_CACHE_DIR) / f"{analysis_ulid}.pdf"
    if cache_path.exists():
        return cache_path.read_bytes()
    
    analysis = await _fetch_analysis(analysis_ulid, session)
    pdf_bytes = await generate_pdf(analysis)
    
    cache_path.parent.mkdir(parents=True, exist_ok=True)
    cache_path.write_bytes(pdf_bytes)
    return pdf_bytes
```

### Fields Explicitly Excluded

- `user_id`, `api_key_id` -- user identity
- `email`, `key_prefix` -- PII
- `mesh_hash` -- could be used for correlation
- `ulid` -- internal ID (short_id is the public identifier)
- `is_public`, `share_short_id` -- internal flags

## 7. Rate Limiting Strategy

### Existing Infrastructure

Phase 2 established `slowapi + Redis` with `_api_key_id()` key function that falls back to IP for unauthenticated requests. The `limiter` instance is in `backend/src/auth/rate_limit.py`.

### New Endpoint Limits

| Endpoint | Auth | Limit | Key |
|----------|------|-------|-----|
| `POST /analyses/{id}/share` | Required | 60/hr, 500/day (existing) | api_key_id |
| `DELETE /analyses/{id}/share` | Required | 60/hr, 500/day (existing) | api_key_id |
| `GET /s/{short_id}` | None | 120/hr | client IP |
| `GET /analyses/{id}/pdf` | Required | 60/hr, 500/day (existing) | api_key_id |

For `GET /s/{short_id}`, the existing `_api_key_id` fallback already handles IP-based limiting when no auth is present. The decorator just needs a different limit string: `@limiter.limit("120/hour")`.

## 8. File System PDF Cache

### Configuration

```python
PDF_CACHE_DIR = os.getenv("PDF_CACHE_DIR", "/data/pdf-cache/")
```

On Fly.io, `/data/` is the standard volume mount point. For local dev, defaults to a local directory.

### Cache Key

`{analysis_ulid}.pdf` -- one file per analysis. Since analyses are immutable (Phase 3), the PDF never needs regeneration unless the template changes.

### Cache Invalidation

Template changes (deployment events) can clear the cache via:
```bash
rm -rf /data/pdf-cache/*.pdf
```

No automated invalidation needed -- analyses are append-only.
