# Phase 9 Plan B: Batch API Endpoints + Coordinator Task + Webhook Delivery Summary

**One-liner:** Five REST endpoints (create/status/items/csv/cancel), arq coordinator with concurrency-limited drip-feed, HMAC-SHA256 webhook delivery with 5-retry exponential backoff.

## What Was Built

### Webhook Service (`webhook_service.py`)
- `sign_webhook_payload`: Stripe-like `t={ts},v1={hmac_hex}` format
- `verify_webhook_signature`: Timing-safe comparison with 5-minute replay window
- `deliver_webhook`: httpx POST with 10s timeout, X-CadVerify-Signature header
- `schedule_webhook_retry`: Exponential backoff [10, 30, 90, 270, 810] seconds with 10% jitter

### Batch Router (`batch_router.py`)
- `POST /api/v1/batch` (202): ZIP upload or S3 reference, enqueues coordinator
- `GET /api/v1/batch/{id}`: Real-time progress via denormalized counters
- `GET /api/v1/batch/{id}/items`: Cursor-paginated with optional status filter
- `GET /api/v1/batch/{id}/results/csv`: Streaming CSV with Content-Disposition
- `POST /api/v1/batch/{id}/cancel`: Skips pending items, 409 for terminal states

### Batch Tasks (`batch_tasks.py`)
- `run_batch_coordinator`: Polls every 2s, drip-feeds items up to concurrency_limit, fires batch.completed webhook
- `run_batch_item`: Reads file from disk/S3, calls `analysis_service.run_analysis`, updates counters atomically, fires item webhook
- `dispatch_webhook`: Delivers webhook and schedules retry on failure

### Worker + App Registration
- Worker: 3 new task functions, max_jobs=12
- App: batch_router registered in FastAPI main

## Security Mitigations

| ID | Threat | Mitigation Applied |
|----|--------|--------------------|
| T-09-03 | Webhook secret in responses | POST /batch response excludes webhook_secret |
| T-09-04 | IDOR on batches | All endpoints filter by user_id; 404 not 403 |
| T-09-05 | Webhook replay attack | Timestamp in HMAC header, 5-minute tolerance |
