# CADVerify Ops Observability — OpenTelemetry tracing + load smoke (2026-07-08)

This documents the ops truth gate added in this packet: opt-in OpenTelemetry
tracing on the costed request path, and a real single-container load smoke. Every
number below comes from a command that was actually run in this container; the
raw captures are the artifacts referenced by path.

## 1. What OpenTelemetry tracing now does

- **Auto-instruments incoming FastAPI requests.** `opentelemetry-instrumentation-fastapi`
  produces the SERVER span per request (`POST /api/v1/validate/cost/demo`) with
  the standard `http.method` / `http.route` / `http.status_code` attributes.
- **Manual stage spans on the costed path** (`POST /api/v1/validate/cost` and its
  `/demo` sibling, which share `_run_cost_decision`):
  - `cost.parse_mesh` — read upload + tessellate/parse to a trimesh + mm rescale
    (attrs: `cadverify.file.suffix`, `cadverify.file.bytes`, `cadverify.units`).
  - `cost.compute` — parent span around the off-loop executor compute.
    - `cost.dfm_analysis` — the DFM / geometry pass (`_run_cost_engine`)
      (attr: `cadverify.face_count`).
    - `cost.should_cost` — the make-vs-buy costing (`estimate_decision`)
      (attr: `cadverify.status`).
    - `cost.ensemble` — only when the opt-in uncertainty ensemble is enabled.
  - `cost.serialize` — glass-box decision serialization (`report_to_dict`).
  The DFM/should-cost spans run inside a thread executor; the OTel context is
  captured on the event loop and re-attached in the worker so they correctly
  nest under `cost.compute` (proven by the parent-id assertions in the tests and
  the captured tree below).
- **Span context / attributes:** the SERVER span carries `cadverify.request_id`
  (the same opaque correlation id `RequestIDMiddleware` already binds to the
  logs) and, for authed calls, `cadverify.org_id` when it is resolved. No new
  PII: both are opaque identifiers, not user content; filenames/CAD bytes are not
  attached (only a suffix + byte count).

### Export modes
- **OTLP/HTTP** to `OTEL_EXPORTER_OTLP_ENDPOINT` (batched) — the production export
  path to a collector/APM.
- **Console** exporter when enabled without an endpoint — spans print to stdout
  for local/dev proof.
- **In-memory** exporter — injected directly by tests and the proof driver for
  deterministic, offline capture.

### The opt-in switch (OFF by default)
Tracing activates only when **`OTEL_TRACING_ENABLED`** is truthy
(`1/true/yes/on`) **or** **`OTEL_EXPORTER_OTLP_ENDPOINT`** is set. `setup_tracing()`
(called once from `main.py`) returns immediately otherwise — importing no
OpenTelemetry, adding no middleware, changing nothing.

### Zero-overhead-when-off proof
`src/obs/tracing.py` never imports `opentelemetry` at module load; all OTel
imports live inside functions on the enabled path. While disabled:
- `tracing.span(...)` returns a single shared `contextlib.nullcontext()` (no
  allocation, no OTel import);
- `capture_context()/attach_context()` are inert no-ops;
- the app imports with `opentelemetry` absent from `sys.modules`.

Verified in-container (tracing unset):

```
$ .venv/bin/python -c "import sys, main; from src.obs import tracing; \
  print('is_active', tracing.is_active()); \
  print('otel_imported', any(m.startswith('opentelemetry') for m in sys.modules)); \
  print('span_is_null', tracing.span('x') is tracing._NULL)"
is_active = False
opentelemetry in sys.modules (should be False) = False
span() off-path is shared nullcontext = True
```

Tests `test_no_spans_and_no_error_when_disabled` and
`test_setup_tracing_noop_when_unconfigured` assert the same, plus that the
identical `POST /validate/cost/demo` still returns `200 OK` with tracing off.

load failure, and is counted as served.

## 3. REAL vs EXTERNAL GATE

**Real (done + proven in this container):**
- Real OpenTelemetry SDK 1.43.0 + FastAPI auto-instrumentation (not a fallback
  tracer — egress was available and the libraries installed).
- Real, correctly-nested spans emitted from a real `/validate/cost/demo` request,
  captured to `otel-trace-2026-07-08.txt`.
- Real in-container load smoke (health, health/deep, costed demo) with real
  p50/p95/p99, throughput, and error rate, captured to `load-smoke-2026-07-08.txt`.
- Default-off / zero-overhead-when-unconfigured, proven by test + the import check.

**External gate (NOT done here; requires infra beyond this container):**
- A live OTel **collector / APM backend** (Honeycomb, Datadog, Grafana Tempo,
  Jaeger). The OTLP/HTTP export path is wired and pinned but was NOT exercised
  against a real collector — no endpoint exists in this container. Proof used the
  in-memory/console exporters only.
- **Production-scale k6** at realistic concurrency against production-grade
  hardware and a warmed cache. k6 is not installed here; the Node smoke is a
  low-concurrency single-container profile only.
- **Distributed, multi-service traces** (context propagation across the API,
  the arq worker, and downstream services). Only the single in-process API
  request path is instrumented here.
- No SLOs/SLA numbers are asserted — the load figures are a smoke, not targets.

## 4. Enabling tracing (operator notes)

Dependencies are optional and kept OUT of the default install (`requirements.txt`
does not list them; the hot path imports zero OTel when off). To enable:

```
pip install -r backend/requirements-otel.txt
# local console/in-memory spans:
OTEL_TRACING_ENABLED=1 uvicorn main:app
# export to a collector:
OTEL_EXPORTER_OTLP_ENDPOINT=http://collector:4318 uvicorn main:app
```

Pinned optional deps (`backend/requirements-otel.txt`):
`opentelemetry-sdk==1.43.0`, `opentelemetry-instrumentation-fastapi==0.64b0`,
`opentelemetry-exporter-otlp-proto-http==1.43.0`.
