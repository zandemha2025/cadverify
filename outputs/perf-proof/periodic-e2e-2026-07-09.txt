THE BUG (measured before the fix)
---------------------------------
Retry-ladder rung 0 = gmsh default 2D algorithm. On a periodic-surface part it
GRINDS then fails; only THEN does the fast MeshAdapt recovery rung run. Direct
timing on this box:
  * ctc_05 rung 0 (default algo, curvature): > 180s and still running (killed) -> FAIL
  * ctc_05 rung 1 (MeshAdapt, uniform):      3.9s -> OK, 56196 nodes
Because the route wraps tessellation in wait_for(ANALYSIS_TIMEOUT_SEC=60s), the
whole ladder exceeded the budget -> the user got a 504; the recovery never landed.

THE FIX (per-rung wall-clock cap in a killable subprocess; parse_pool)
----------------------------------------------------------------------
Each ladder rung is a separately-submitted, separately-timed unit. rung 0 runs on
the shared warm pool with a 24s cap; its periodic GRIND is abandoned at the cap and
the recovery rungs run in dedicated single-worker spawn subprocesses, HARD-KILLED
(SIGKILL) at their caps. gmsh.mesh.generate is an uninterruptible C call, so only a
subprocess SIGKILL reclaims a grinding rung.

MECHANISM NOTE (honest)
-----------------------
Mechanism #1 (cheap periodic pre-detection to SKIP rung 0) was investigated and
REJECTED on this box: the only cheap gmsh signal (surface parametric periodicity /
wrap) is ANTI-correlated with the failure. The FAST part ftc_11 has 4 periodic
(cylinder) surfaces that mesh fine in ~1.4s, while the problematic ctc_02/04/05
report ZERO by that test. Diverting on periodicity would (a) wrongly divert fast
parts -> break byte-identity, and (b) miss the real failure. gmsh's Python API
exposes no reliable cheap signal for the SPECIFIC "Impossible to mesh periodic
surface N" condition. Per the task's explicit fallback, we rely on Mechanism #2
(per-rung wall-clock cap in a killable subprocess), which is mechanism-agnostic and
demonstrably works here.
